Executive summary

Over the 20-week period, 20 separate outages were recorded, totalling roughly 24 hours 50 minutes of downtime. Three causes dominate. DNS-related failures are the most frequent problem (8 of 20 events) and have become the recurring issue since April. Expired SSL certificates produced the two single longest outages and the most lost minutes per event. Capacity problems on the L-sized server were the early pain point but appear resolved after the upgrade to an XL instance at the end of March.

Every issue below has a clear fix, and most are straightforward for the team to action. One choice needs a decision from leadership — how to renew the SSL certificates — where the development team recommends Option 1. It is summarised in Problem → Fix.

How the incidents connect

The DDoS and DNS categories are linked. The DDoS event on 5 March prompted an architecture change: traffic was moved from connecting directly to the EC2 instance to routing through Amazon CloudFront, in order to absorb attack traffic and gain better control over requests to the web app. That migration required repointing DNS from EC2 to CloudFront — and is the origin of the recurring DNS failures described below.

The capacity crashes are a separate, now-closed thread, resolved by the upgrade from an L to an XL instance at the end of March. They are not related to the CloudFront migration.

Capacity crashes stopped after the XL upgrade

All four “server dead (L size)” crashes occurred on or before 20 March. None have recurred since the instance was resized to XL at the end of March — the upgrade appears to have addressed the under-provisioning.

DNS failures trace back to the CloudFront migration

After moving the app behind CloudFront, DNS had to be repointed from EC2 to CloudFront. KMUTT CC (the DNS administrator) was asked to add either an A record to the CloudFront domain or a CNAME, and chose to add an A record. Because CloudFront serves from a rotating pool of edge IP addresses, a fixed A record can fail whenever those addresses change — consistent with the recurring early-morning outages (~04:00–07:30) seen since the migration. A CNAME or alias record would follow CloudFront's current addresses automatically.

SSL expiries are rare but very costly

The two certificate-expiry events (11 Apr and 9 Jun) caused 5h 08m and 3h 29m of downtime — the worst outages in the period. Both the CloudFront and ALB certs are Let's Encrypt imported into ACM, and imported certs don't auto-renew — so a single missed manual re-import took both down at once. Fully preventable with native ACM certificates.

SSLAction needed

Expired TLS certificates (CloudFront + ALB)

8h 37m · 2 events · longest outages

Both certs are Let's Encrypt imported into ACM — one for CloudFront, one for the ALB. Imported certs don't auto-renew in ACM, so when the manual 90-day re-import was missed, both expired together.

Recommended fix

Replace them with native ACM certificates, which auto-renew on both CloudFront and the ALB (KMUTT CC adds the DNS validation record once). A renewal script for Let's Encrypt is a fallback only — it rebuilds the same fragility. See actions 1 and 2.

DNSAction needed

Wrong DNS record type after CloudFront move

9h 42m · 8 events · most frequent

KMUTT CC added an A record instead of a CNAME for CloudFront. A fixed A record breaks when CloudFront rotates its edge IPs.

Recommended fix

Ask KMUTT CC to change the record from A to CNAME (or alias). If outages continue afterwards, investigate the root cause jointly with KMUTT CC.

How should we renew the TLS certificates?

Dev recommendation

Option 1 · AWS-managed certificates

What it means
Switch CloudFront and the load balancer to certificates that AWS issues and renews for us.
Effort
One-time setup, then nothing. Requires KMUTT CC to add one DNS record.
Ongoing risk
None — renews automatically, so expiries can't recur.
Depends on
KMUTT CC adding and keeping a single DNS record.
Alternative

Option 2 · Keep current certs + renewal script

What it means
Build a program that renews the existing Let's Encrypt certificates and reloads them into AWS automatically.
Effort
Build and maintain a custom script covering both endpoints.
Ongoing risk
A moving part that can fail quietly — this is what caused the two outages.
Depends on
The script running reliably every cycle; still needs a validation step.

Recommendation: Option 1. Neither option has any licensing cost — the difference is engineering effort and risk. Option 1 removes the expiry risk permanently for a one-time setup; Option 2 rebuilds the fragility behind these outages. Either way, the only thing needed from leadership is approval to proceed; Option 1 also needs a one-off request to KMUTT CC for a DNS record.

Technical: Option 1 = native ACM certificates (us-east-1 for CloudFront, the ALB's region for the load balancer), DNS-validated; one validation CNAME covers both since they share a domain and account. Option 2 = scheduled job (e.g. Lambda + EventBridge) that re-imports to the same ACM ARNs, plus a 14-day expiry alarm as a backstop.

Other recommended actions

  1. Correct the DNS record type with KMUTT CC. Contact KMUTT CC and request that the CloudFront record be changed from an A record to a CNAME (or alias), so DNS resolves to CloudFront's current edge addresses instead of a fixed IP.
  2. If outages persist, investigate jointly with KMUTT CC. Should DNS failures continue after the record change, trace the root cause together with KMUTT CC rather than in isolation — the records are managed on their side.
  3. Confirm the XL instance is the right long-term size. The upgrade appears to have fixed the capacity crashes — keep monitoring CPU/memory headroom and set autoscaling or alerting before load reaches the previous failure threshold.
  4. Record a root cause for every incident. Two events (12 Feb, 12 Mar) have no recorded cause. Capturing cause, detection method, and resolution makes future trend analysis far more reliable.
  5. Build on the CloudFront DDoS protection already in place. The CloudFront migration is itself the DDoS mitigation, so no separate CDN is needed. If not already enabled, consider layering AWS WAF / Shield on the distribution for rate-limiting and additional attack coverage.
